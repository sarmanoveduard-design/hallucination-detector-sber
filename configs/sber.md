# Guardian of Truth: Детекция галлюцинаций LLM
## Введение в проблематику
Большие языковые модели (LLM) активно внедряются в промышленную эксплуатацию. В инфраструктуре Сбера уже работают сотни ИИ‑агентов, ежедневно обрабатывающих миллионы клиентских обращений. Несмотря на высокое качество генерации, LLM обладают фундаментальным ограничением — они могут галлюцинировать.

Галлюцинацией называется ситуация, когда модель генерирует уверенное, но фактически некорректное или не подтвержденное утверждение. Такие ошибки могут вводить пользователей в заблуждение, снижать доверие к системе и создавать репутационные и финансовые риски.

Задача — разработка системы автоматической детекции галлюцинаций, которая сможет обнаруживать потенциально ложные ответы до того, как они будут отправлены пользователю.
## Типы галлюцинаций
Галлюцинации в LLM можно классифицировать следующим образом:
Фактологическая галлюцинация — модель генерирует утверждение, противоречащее общеизвестным фактам (например, «Пётр I жил в XIV веке»).
Контекстуальная галлюцинация — информация присутствует во входном контексте, однако модель игнорирует её или искажает.
Логическая галлюцинация – модель выдает неверные логические суждения и делает некорректные выводы.
Галлюцинация в вычислениях, формулах, арифметике, единицах измерения.
Галлюцинация в инструкциях – модели в системном промпте указаны инструкции которые она игнорирует.
Конфабуляция – вместо того чтобы ответить «не знаю», модель додумывает ответ на вопрос, на который либо не существует ответа, либо она не может ответить ввиду knowledge cut-off.
В рамках данной рассматривается только фактологические галлюцинации!
## Входные данные
Для оценки решений будет использоваться специализированный бенчмарк. Примеры правильных ответов и галлюцинаций в бенчмарке генерируются моделью ai-sage/GigaChat3-10B-A1.8B-bf16.

Бенчмарк включает:
- корректные ответы
- фактологические галлюцинации
Участникам хакатона будет предоставлена preview версия бенчмарка, предназначенный для разработки архитектуры детектора и предварительной оценки качества и скорости работы решений.
При этом финальный тестовый бенчмарк не будет доступен участникам во время хакатона.
Он будет опубликован после завершения соревнования и использован для окончательной оценки решений.
## Критерии оценивания
Помимо точности детекции важным критерием является скорость работы алгоритма. Мы вводим ограничение на время детекции: 500 ms на один ответ, а также сообщаем, что проверка решений будет проводиться на 1 GPU A100 80GB.
Если используются методы, требующие forward pass какой либо модели, то время этого forward pass не учитывается только в случае если используется модели ai-sage/GigaChat3-10B-A1.8B-bf16. Если метод подразумевает анализ градиентов, то время на backward pass будет учитываться вне зависимости от модели
Частью задачи является сбор данных на основе которых будет: обучаться модель / функционировать архитектура. Обучаться на предоставленном preview бенчмарка нельзя, он предоставлен как публичный тест для вашего удобства разработки архитектуры о оценки своих решений. Ограничений на источники данных нет, однако встраивание внешних API в архитектуру не допускается.
Также в основном пайплайне необходимо реализовать замер времени работы модели детекции галлюцинаций. Как уже сказано ранее разрешается не учитывать 1 forward pass модели ai-sage/GigaChat3-10B-A1.8B-bf16 на последовательности токенов prompt+response, например для снятии проб со скрытых слоев (см. baseline.ipynb)
Финальная оценка будет проводиться по PR-AUC. В случае решения с одинаковыми метриками лучшее будет определяться по скорости работы алгоритма.
## Классификация методов детекции
Методы детекции галлюцинаций можно условно разделить на три группы:

## Внешняя верификация (External)
## Внутренние сигналы модели (Internal)
## Методы оценки неопределенности (Uncertainty)
Ниже мы приводим некоторые общеизвестные методы, которые можно применять в ходе построения решения. (Но не обязательно!)

| Категория | Метод | Описание |
| --- | --- | --- |
| External | Retrieval-based fact checking | Извлечение утверждений и проверка через поиск документов |
| External | NLI-based verification | Проверка логического отношения между контекстом и утверждением |
| External | Claim extraction + verification | Разбиение ответа на утверждения и проверка каждого |
| External | Context–response classifier | Обучается классификатор на паре (prompt + response, тип галлюцинации/ее отсутствие) |
| Internal | Hidden state probing | Использование hidden states для обучения классификатора |
| Internal | Residual stream analysis | Анализ динамики внутренних представлений по слоям |
| Internal | Layer disagreement | Сравнение распределений токенов разных слоев |
| Internal | Logit lens | Применение LM head к промежуточным слоям |
| Internal | Activation statistics | Анализ статистик активаций скрытых состояний |
| Uncertainty | Token log probability | Использование вероятностей токенов ответа |
| Uncertainty | Entropy / surprisal | Использование энтропии распределения токенов |
| Uncertainty | Self-consistency | Генерация нескольких ответов и сравнение согласованности |

Важно! Мы не ограничиваем этим списком вашу свободу выбора подхода к решению этой задачи, а наоборот поощряем исследовать новые, нестандартные методы в борьбе с галлюцинациями.
## Итоговая цель
Цель участников — разработать систему детекции галлюцинаций, которая демонстрирует высокий баланс между точностью и скоростью работы.

Решение должно быть применимо в production‑среде, а именно соответствовать ограничениям по времени обработки и используемым ресурсам.
